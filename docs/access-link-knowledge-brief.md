# ACCESS LINK 이해 정리

작성일: 2026-07-23

## 1. 장비 한 줄 정의

ACCESS LINK는 앱, 서버, 태블릿 같은 소프트웨어를 현장의 실제 장비와 연결해 주는 범용 인터페이스 장비다.

출입통제는 대표 사용 사례 중 하나이며, 장비 자체는 출입문 전용 제품이 아니다.

## 2. 언제 사용하는 장비인가

ACCESS LINK는 다음 상황에서 사용한다.

- 앱은 이미 있지만 카드리더기, 릴레이, 센서, 산업 장비와 직접 연결해야 할 때
- 프로젝트마다 RS-232, RS-485, Wiegand, Digital Input, Relay 같은 통신 규격이 달라서 대응이 어려울 때
- PCB 설계, 펌웨어 개발, 인증 절차 없이 빠르게 현장 장비 연동 데모를 만들어야 할 때
- 소프트웨어 회사, SI 회사, IoT 스타트업, 교육·연구기관, 무인화·스마트공간 사업자가 하드웨어 팀 없이 현장 장비를 제어해야 할 때
- 출입통제, 무인매장, 스마트오피스, 락커, 스마트팜, 산업 IoT 같은 현장 장비 제어가 필요할 때

## 3. 장비 역할

ACCESS LINK는 소프트웨어 쪽 통신과 현장 장비 쪽 신호를 서로 변환한다.

소프트웨어 쪽:

- Ethernet
- USB Type-C

현장 장비 쪽:

- Wiegand 카드리더기 입력·출력
- RS-232
- RS-485
- Digital Input
- Relay Output

즉 앱이 직접 Wiegand나 릴레이를 처리하는 것이 아니라, 앱은 ACCESS LINK에 명령을 보내고 ACCESS LINK가 현장 장비 신호로 변환한다.

## 4. 출입통제 관점에서의 의미

출입통제 모드에서 ACCESS LINK는 다음 흐름의 중간 장비가 된다.

```text
사용자 인증 앱
→ ACCESS LINK
→ Wiegand 리더기 / Relay Output / Digital Input
→ 도어락, Exit 버튼, 센서
```

이때 앱이 담당하는 것은 다음이다.

- 카드 또는 얼굴이 등록된 사용자인지 판단
- 인증 성공 시 어떤 Relay를 출력할지 결정
- ACCESS LINK로 릴레이 제어 명령 전송
- 인증 성공, 실패, 릴레이 송신, 입력 감지 로그 저장

ACCESS LINK가 담당하는 것은 다음이다.

- Wiegand 입력 수신
- RS-232/RS-485 송수신
- Digital Input 상태 전달
- Relay 0/1 출력
- Ethernet 또는 USB를 통한 앱·서버와의 통신

## 5. 출입통제에만 고정하면 안 되는 이유

사내 마케팅 자료는 ACCESS LINK를 출입통제 제품으로만 설명하지 않는다.

자료 기준으로 출입통제는 대표 사용 사례이며, 실제 포지션은 앱과 현실 장비를 연결하는 플랫폼이다.

따라서 앱 기본 용어는 다음처럼 장비 기능 중심이어야 한다.

- 문 설정보다 릴레이 설정
- 1번 문, 2번 문보다 Relay 0, Relay 1
- 문 개방보다 릴레이 출력 또는 출력 실행
- 출입통제 모드에서만 정문, 후문, 문 열림 같은 표현 사용

## 6. 앱이 제공해야 하는 기능

앱은 크게 세 영역으로 나누는 것이 맞다.

### 6.1 운영 화면

현장 사용자가 보는 화면이다.

사용 사례에 따라 화면이 달라질 수 있다.

- 출입통제 모드: 얼굴 인증, 카드 인증, 인증 성공·실패 표시
- 릴레이 제어 모드: Relay 0/1 직접 출력
- 데모 모드: 앱 명령으로 실제 장비가 움직이는 시연

### 6.2 관리자 설정

관리자 또는 설치자가 현장 조건에 맞게 장비를 설정하는 화면이다.

- 장비 연결 설정
- 릴레이 설정
- 입력 설정
- Wiegand 카드리더 설정
- RS-232/RS-485 통신 설정
- 사용자, 카드, 얼굴 등록
- 인증 성공 시 실행할 Relay 설정
- 로그 확인

### 6.3 고급 진단

설치자와 개발자가 장비 상태를 확인하는 화면이다.

- USB-C 연결 상태
- Ethernet 연결 상태
- Relay 0/1 테스트
- Wiegand 입력·출력
- RS-232 송수신
- RS-485 송수신
- Digital Input 0/1 상태
- Raw Packet 로그

## 7. 연결 방식 이해

### 7.1 USB-C

USB-C는 근거리 연결, 초기 설정, 현장 진단, 개발 테스트에 적합하다.

현재 앱은 CH340 USB Serial 기준으로 ACCESS LINK와 통신한다.

현재 기준 상태:

- USB 장치 감지 가능
- USB 권한 요청 가능
- 자동 연결은 꺼져 있음
- 관리자가 수동 연결해야 시리얼 포트를 연다

USB-C 연결 시 장비가 반복 재부팅되는 경우, 앱 패킷 문제보다 USB-C 전원 역할 협상, 케이블, OTG 방식, ACCESS LINK 전원 구조를 먼저 확인해야 한다.

### 7.2 Ethernet

제품 포지션상 Ethernet은 운영 환경에서 중요한 연결 방식이다.

Ethernet으로 가능해야 하는 기능:

- Relay 0/1 제어
- Wiegand 입력 실시간 수신
- RS-232 송수신
- RS-485 송수신
- Digital Input 0/1 실시간 수신
- 장비 연결 상태 확인
- 통신 로그 기록

아직 확인이 필요한 정보:

- ACCESS LINK의 Ethernet 통신 방식
- TCP, UDP, HTTP 중 어떤 방식인지
- 기본 IP 또는 DHCP 사용 여부
- 포트 번호
- USB Serial과 같은 SHAL 패킷을 Ethernet으로 그대로 보내는지
- 장비 검색 기능이 있는지
- 이벤트를 서버처럼 push하는지, 앱이 poll 해야 하는지

## 8. 프로토콜 기준 핵심 명령

현재 확인된 SHAL-1000 프로토콜 명령은 다음과 같다.

| Command | 이름 | 의미 |
| --- | --- | --- |
| `0x00` | `SET_RELAYCONTROL` | 릴레이 제어 |
| `0x01` | `SET_SERIALSEND` | RS-232/RS-485 데이터 출력 |
| `0x02` | `SET_WIEGANDOUT` | Wiegand 출력 |
| `0x09` | `GET_WIEGANDINPUTDATA` | Wiegand 입력 데이터 수신 |
| `0x10` | `GET_RECVDATA0` | RS-232 수신 |
| `0x11` | `GET_RECVDATA1` | RS-485 수신 |
| `0x12` | `GET_INPUTPORT0` | Input Port 0 상태 |
| `0x13` | `GET_INPUTPORT1` | Input Port 1 상태 |

주의할 점:

- Wiegand 입력은 명령 목록 기준 `0x09`로 유지한다.
- 문서의 Wiegand 상세 페이지에는 응답 Command가 `0x10`으로 표기되어 있으나, `0x10`은 RS-232 수신과 충돌한다.
- 실제 장비 로그에서 근거가 확인되기 전까지 임의로 매핑을 바꾸지 않는다.

## 9. 릴레이 이해

프로토콜 기준 릴레이 제어 값은 다음과 같다.

- `UseRelay = 0`: Relay 0 + Relay 1 동시 제어
- `UseRelay = 1`: Relay 0
- `UseRelay = 2`: Relay 1

앱 UX 기준:

- 일반 관리자 설정에는 Relay 0, Relay 1만 기본 노출한다.
- Relay 0+1은 실수로 두 출력을 동시에 켤 수 있으므로 일반 기능에 노출하지 않는 것이 좋다.
- 필요하면 고급 진단 또는 공장 테스트 용도로만 숨겨서 제공한다.

## 10. 앱 수정 방향

앞으로 앱은 출입통제 전용 표현을 줄이고 범용 장비 제어 기준으로 정리한다.

우선순위:

1. 현재 기준 상태 고정
2. 문 설정 표현을 릴레이 설정으로 변경
3. Relay 0/1 설정 화면 정리
4. 장비 연결 화면에서 USB-C와 Ethernet을 분리
5. 고급 진단 화면을 Wiegand, RS-232, RS-485, Digital Input, Relay로 분리
6. 인증 성공 시 실행할 Relay 설정으로 문구 변경
7. Digital Input 0/1을 범용 입력으로 설정
8. 카드와 얼굴을 사용자 모델로 통합
9. 운영 모드별 사용자 화면 정리
10. 로그와 감사 이력 정리

## 11. 부장님 또는 제조 담당자에게 확인해야 할 질문

Ethernet 관련:

- ACCESS LINK Ethernet 통신은 TCP, UDP, HTTP 중 무엇인가?
- 기본 IP가 있는가, DHCP로 받는가?
- 기본 포트 번호는 무엇인가?
- USB Serial과 동일한 SHAL 패킷을 Ethernet으로 그대로 보내면 되는가?
- Wiegand, RS-232, RS-485, Digital Input 이벤트를 Ethernet으로 실시간 push 하는가?
- 앱이 장비를 자동 검색할 수 있는 discovery 기능이 있는가?

USB-C 관련:

- USB-C는 데이터 통신과 전원 공급을 동시에 지원하는가?
- 폰 C-to-C 직결을 공식 지원하는가?
- OTG 어댑터 또는 전원 분리형 허브 사용을 권장하는가?
- 모바일 충전 지원 문구가 폰을 충전한다는 의미인지, 폰에서 장비 전원을 공급할 수 있다는 의미인지 확인이 필요하다.

전원·결선 관련:

- ACCESS LINK의 표준 운영 전원은 DC 12V가 맞는가?
- Wiegand 리더기 전원은 ACCESS LINK에서 공급하는 구조인가, 외부 전원 병렬 연결이 권장되는가?
- Relay Output은 dry contact 기준인가?
- DC 락이나 솔레노이드 부하 연결 시 보호 다이오드 또는 서지 보호가 필요한가?

## 12. 현장 테스트 체크리스트

USB-C 테스트:

- 앱 종료 상태에서 폰과 ACCESS LINK C-to-C 연결
- ACCESS LINK 전원 LED가 꺼졌다 켜지는지 확인
- 앱 실행 후 USB 장치가 감지되는지 확인
- USB 권한 요청이 정상 표시되는지 확인
- 수동 연결 버튼을 누를 때만 시리얼 연결되는지 확인

Ethernet 테스트:

- ACCESS LINK가 IP를 받는지 확인
- 같은 네트워크에서 ping 또는 TCP 연결이 가능한지 확인
- 지정 포트로 연결 가능한지 확인
- SHAL 패킷 송수신이 가능한지 확인
- Wiegand 입력, Digital Input, RS-232/485 이벤트가 실시간 수신되는지 확인

릴레이 테스트:

- Relay 0 단독 출력
- Relay 1 단독 출력
- 출력 시간 1초, 3초, 5초 확인
- 지속 ON과 OFF는 고급 진단에서만 확인
- 물리 장비 동작 성공과 앱의 송신 성공을 구분해 기록

## 13. 현재 결론

ACCESS LINK 앱은 출입통제 앱 하나로 좁히면 안 된다.

기본은 ACCESS LINK의 7종 인터페이스를 관리하고 테스트하는 범용 앱이어야 하며, 출입통제는 그 위에 얹는 운영 모드 중 하나로 설계해야 한다.

따라서 앞으로의 앱 개발 기준은 다음이다.

- 장비 기본 용어는 Relay, Input, Wiegand, RS-232, RS-485, Ethernet, USB를 사용한다.
- 출입통제 표현은 출입통제 모드에서만 사용한다.
- USB-C와 Ethernet은 둘 다 지원하되, 운영 연결은 Ethernet 중심으로 확장한다.
- 사용자가 실수로 실제 장비를 동작시키는 위험 기능은 고급 진단 안에 숨긴다.
- 실제 장비 상태를 알 수 없는 경우 송신 성공과 물리 동작 성공을 명확히 구분한다.
