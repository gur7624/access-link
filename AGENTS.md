# AGENTS.md

## 기본 원칙

- 모든 사용자 응답과 작업 진행 메시지는 한국어로 작성한다.
- 작업 진행 메시지에는 상세한 내부 추론을 공개하지 않고, 현재 작업·확인한 근거·검증 결과만 간결하게 공유한다.
- 변경 전 관련 파일과 영향 범위를 확인한다.
- 요청 범위 내 기존 파일 수정은 허용한다. 다만 파일 삭제, 대규모 덮어쓰기, 무관한 리팩토링은 사용자 승인 없이 하지 않는다.
- 사용자가 요청하지 않은 의존성 추가, 패키지 구조 변경, public API 변경은 피한다.
- 추측으로 수정하지 않는다. 원인을 확인할 수 없으면 먼저 분석하고, 위험한 변경은 사용자에게 확인한다.
- 보안·데이터 손실·실제 하드웨어 결선에 영향을 주는 불확실한 사항은 임의로 진행하지 않고 확인한다.
- 이 프로젝트는 프로덕션을 전제로 한다. 임시 우회 코드나 테스트용 보안 취약 코드를 작성하지 않는다.

## 작업 방식

1. 요구사항과 현재 동작을 짧게 정리한다.
2. 관련 코드와 `docs/` 문서를 UTF-8로 확인한다.
3. 영향이 가장 작은 방식으로 수정한다.
4. 변경 기능과 직접 영향받는 흐름을 검증한다.
5. 오류가 발생하면 작업 단계, 오류 메시지, 원인, 해결 또는 남은 조치를 기록한다.

## 문서 규칙

- `docs/` 폴더가 있으면 관련 Markdown 문서를 먼저 확인하고 문서의 규칙을 따른다.
- Markdown은 UTF-8 기준으로 읽고 저장한다. 인코딩이 불명확한 파일은 원본을 임의로 변환하지 않는다.
- 기능 변경 시 프로젝트의 변경 이력 문서가 있으면 날짜와 변경 내용을 함께 갱신한다.

## 보안 및 로그

- PIN, 카드 식별자, 얼굴 특징값, 출입 기록은 민감 정보로 취급한다.
- 인증·권한·저장·네트워크 코드에서 평문 노출, 무제한 재시도, 검증 없는 입력, 임의의 릴레이 제어를 사용하지 않는다.
- 임시 디버그 로그와 운영 감사 로그를 구분한다. 문제 해결 후 임시 디버그 로그는 제거하되, 인증·출입·릴레이 송신 감사 로그는 보존한다.
- 실제 장비 상태를 확인할 수 없는 경우 송신 성공과 물리적 동작 성공을 구분해 표시한다.

## Android 프로젝트 검증

이 저장소의 기본 검증 순서는 다음과 같다.

```text
./gradlew test
./gradlew lintDebug
./gradlew assembleDebug
./gradlew connectedDebugAndroidTest
```

- 환경 문제로 명령을 실행하지 못하면 코드 오류와 환경 오류를 구분해 보고한다.
- 테스트 실패 상태에서는 작업을 완료 처리하지 않는다.
- 카메라·USB·시리얼·릴레이처럼 실제 장비가 필요한 기능은 자동 테스트 결과와 현장 테스트 결과를 구분한다.

## Go 프로젝트 규칙

Go 파일이나 Go 모듈이 있는 작업에만 다음 규칙을 적용한다.

- `gofmt`, `go vet`, `go test` 기준을 따른다.
- 에러를 무시하지 않고 명확하게 처리한다.
- 불필요한 전역 변수와 큰 인터페이스를 피한다.
- 동시성 코드의 race condition을 점검한다.
- `context`는 요청 흐름에 맞게 전달한다.
- handler, service, repository의 책임을 섞지 않는다.

## 웹 검색 기준

- 저장소 코드와 로컬 문서를 먼저 확인한다.
- Android 정책, 라이브러리 버전, 외부 장비 사양, 법률·보안 기준처럼 변경 가능성이 있는 내용만 필요할 때 웹 검색한다.
- 웹 검색을 사용하면 최종 응답에 확인한 출처를 함께 표시한다.

## 최종 응답

최종 응답에는 작업 성격에 따라 다음 내용을 포함한다.

- 원인 또는 작업 요약
- 변경한 파일
- 핵심 변경 내용
- 검증 결과
- 남은 위험 또는 추가 확인 사항

코드 리뷰 요청에서는 발견 사항을 심각도 순으로 먼저 제시하고, 파일과 라인을 함께 표시한다. 수정하지 않은 경우에도 그 사실을 명확히 밝힌다.
